Total Recall: Engineering the Five Memory Systems Every Agentic AI Needs

An LLM without memory is a brilliant amnesiac — extraordinary reasoning trapped in a perpetual present tense. This essay dissects the five memory systems that transform stateless models into agents that learn, adapt, and accumulate expertise: short-term, long-term, episodic, semantic, and procedural memory.

SJ
Sushant Joshi
Principal Engineer · AI Systems
· April 18, 2026  ·  32 min read

Consider what happens when you ask a senior engineer to debug a production outage. They don't start from scratch. They recall similar incidents (episodic memory), pull up relevant system architecture (semantic memory), apply their debugging playbook (procedural memory), hold the current stack trace in focus (short-term memory), and integrate all of this into an evolving mental model (long-term memory). Now consider what happens when you ask an LLM to do the same thing: it reads the prompt, generates a response, and forgets everything the moment the conversation ends.

This is the fundamental bottleneck in agentic AI. Not reasoning ability — modern LLMs reason remarkably well. Not tool use — we've solved that with function calling and MCP. The bottleneck is memory: the ability to accumulate, organize, retrieve, and apply experience across time, tasks, and contexts.

Every agent system we've explored in this series — from autonomous software engineering to security operations to drug discovery — has confronted the same architectural question: how does the agent remember? In Essay 01, we introduced working memory and episodic memory as components of a code agent. In Essay 03, we built a hypothesis tree that functioned as a specialized scientific memory. But we treated memory as a feature of each domain. It's time to treat it as the foundation.

"Memory is not a feature you bolt onto an agent. Memory is what makes a system an agent instead of a sophisticated auto-complete."

1. Why Agentic Memory Is Different from Traditional Storage

Databases store data. Caches speed up retrieval. Neither constitutes memory in the cognitive sense. Agentic memory must do four things that traditional storage systems never had to:

🧠
Context-Dependent Recall
The same fact must surface or stay hidden depending on what the agent is currently doing — just like how you remember your grandmother's phone number when planning a visit, not during a code review.
⏳
Temporal Decay & Reinforcement
Recent experiences should be more accessible than distant ones, unless the distant ones were reinforced through repeated use — the spacing effect from cognitive science.
🔀
Cross-Modal Integration
An agent must combine memories from code, conversations, documents, tool outputs, and error logs into a unified understanding — not siloed data stores.

Human cognitive science identifies five distinct memory systems, each serving a different function. The insight that drives this essay is that agentic AI needs all five — and the engineering challenge is implementing each one in a way that plays to the strengths (and works around the limitations) of transformer-based language models.

⚡
Short-Term
The scratchpad — what the agent is actively holding in focus right now
🗄️
Long-Term
The archive — durable knowledge that persists across sessions and tasks
📖
Episodic
The autobiography — specific events the agent experienced and their outcomes
🌐
Semantic
The encyclopedia — facts, concepts, and relationships independent of when they were learned
⚙️
Procedural
The muscle memory — learned sequences of actions for accomplishing specific tasks

2. The Unified Memory Architecture

Before we deep-dive into each memory type, let's see how they fit together. The architecture below shows five memory systems feeding into a central Memory Router — a retrieval orchestrator that decides which memories to surface for a given task context, subject to the hard constraint of the LLM's context window.

Agentic Memory Architecture — Five Systems, One Context Window
Fig 1. Five memory systems feed through a Memory Router into a budget-constrained context window. Dashed lines show write-back paths where the LLM's outputs update each memory system after inference.

The critical insight in this architecture is the Memory Router. The agent doesn't dump all memories into the context window — that would exhaust the token budget in seconds. Instead, the router takes the current task context, formulates retrieval queries against each memory system in parallel, ranks the results by relevance, and packs only the highest-value memories into the available token budget. This is the equivalent of your brain's attentional spotlight: not everything you know is conscious at once, but the right things surface when you need them.

3. Short-Term Memory: The Context Window as RAM

Short-term memory is the only memory system that comes "for free" with an LLM — it is the context window. Every token in the prompt and the conversation history is immediately accessible to the model's attention mechanism. No retrieval latency, no embedding lookup, no database query. This is the agent's working scratchpad.

But "free" is misleading. The context window is the most expensive memory per byte in the entire system. At current pricing, filling a 200K-token context window with Claude costs roughly $60 in input tokens alone. And once you exceed the window, information doesn't gracefully degrade — it vanishes entirely.

Architectural Insight

Short-term memory management is fundamentally a bin-packing problem. You have a fixed-capacity container (context window) and variable-sized items (conversation turns, tool outputs, retrieved memories) that must be packed to maximize the value of what fits. The engineering challenge is deciding what to evict when space runs out.

The Sliding Window with Priority Eviction

The naive approach — first-in, first-out eviction — fails catastrophically. Early conversation turns often contain the user's core intent and constraints. Evicting them causes the agent to lose the plot entirely. A production-grade STM manager must implement priority-based eviction that considers the semantic importance of each item, not just its age.

Python 
from dataclasses import dataclass, field
from enum import IntEnum
from typing import List, Optional
import tiktoken

class Priority(IntEnum):
    """Eviction priority — higher values are evicted first."""
    SYSTEM = 0       # Never evict: system prompt, tool definitions
    PINNED = 1       # User-pinned: explicit "remember this" instructions
    GOAL = 2         # Current task goal and constraints
    RECENT = 3       # Last N conversation turns (recency window)
    RETRIEVED = 4    # Memories injected by the Memory Router
    TOOL_OUTPUT = 5  # Raw tool outputs (often verbose, summarizable)
    STALE = 6        # Older conversation turns outside recency window

@dataclass
class ContextItem:
    content: str
    priority: Priority
    token_count: int
    turn_index: int           # Conversation position
    summary: Optional[str] = None  # Compressed version if available
    summary_tokens: int = 0

class ShortTermMemory:
    """Manages the LLM's context window as a priority-eviction buffer."""

    def __init__(self, max_tokens: int = 128_000, recency_window: int = 6):
        self.max_tokens = max_tokens
        self.recency_window = recency_window   # Keep last N turns at RECENT priority
        self.items: List[ContextItem] = []
        self.encoder = tiktoken.get_encoding("cl100k_base")

    def add(self, content: str, priority: Priority) -> None:
        tokens = len(self.encoder.encode(content))
        item = ContextItem(
            content=content,
            priority=priority,
            token_count=tokens,
            turn_index=len(self.items),
        )
        self.items.append(item)
        self._rebalance()

    def _rebalance(self) -> None:
        """Evict lowest-priority items until we're under budget."""
        # Step 1: Update priorities based on recency
        max_turn = max(i.turn_index for i in self.items)
        for item in self.items:
            if item.priority == Priority.RECENT:
                if max_turn - item.turn_index >= self.recency_window:
                    item.priority = Priority.STALE

        # Step 2: Calculate total tokens
        total = sum(i.token_count for i in self.items)
        if total <= self.max_tokens:
            return

        # Step 3: Evict from highest priority value (lowest importance) first
        eviction_order = sorted(
            self.items,
            key=lambda i: (-i.priority, -i.turn_index)
        )

        for item in eviction_order:
            if total <= self.max_tokens:
                break
            if item.priority <= Priority.PINNED:
                continue  # Never evict system or pinned items

            # Try to compress before fully evicting
            if item.summary and item.priority == Priority.TOOL_OUTPUT:
                saved = item.token_count - item.summary_tokens
                item.content = item.summary
                item.token_count = item.summary_tokens
                total -= saved
            else:
                total -= item.token_count
                self.items.remove(item)

    def pack(self) -> str:
        """Assemble the final context window content."""
        ordered = sorted(self.items, key=lambda i: (i.priority, i.turn_index))
        return "\n\n".join(item.content for item in ordered)
Production Warning

Token counting with tiktoken adds ~2ms per item. For high-throughput agents processing hundreds of items, pre-compute token counts at ingestion time and cache them — don't recount on every rebalance cycle.

Summarization as Compression

The most powerful technique for extending effective STM capacity is progressive summarization. When a tool output consumes 3,000 tokens but its essential information fits in 200, a summarization pass can recover 93% of that budget. The tradeoff is latency (an LLM call to summarize) and fidelity (summaries lose detail). In practice, tool outputs and older conversation turns are excellent summarization candidates. The current task goal and user constraints should never be summarized — they need full fidelity.

This is exactly what systems like Claude Code do: when the conversation history grows too long, earlier messages are compressed into summaries that preserve the essential decisions, constraints, and context while freeing token budget for new information.

4. Long-Term Memory: The Persistent Archive

If short-term memory is RAM, long-term memory is the hard drive. It persists across conversations, sessions, and even agent restarts. Every piece of information the agent has ever processed is a candidate for long-term storage — but storing everything is neither practical nor useful. The engineering challenge is deciding what to consolidate from short-term into long-term memory, and how to retrieve it efficiently when needed.

Scenario: The Returning User

A developer opens Claude Code on Monday to refactor an authentication module. They discuss the codebase structure, make architectural decisions, and establish coding conventions. On Wednesday, they return to continue the work. Without long-term memory, the agent has no idea who they are, what was decided, or what conventions to follow. With it, the agent picks up exactly where they left off — recalling that the team uses dependency injection, that the auth module touches three services, and that the developer prefers explicit error types over exceptions.

The Consolidation Pipeline

Long-term memory consolidation mirrors how the human brain transfers information from hippocampal short-term storage to neocortical long-term storage during sleep. For an agent, "sleep" is the period between sessions. The consolidation pipeline runs after each conversation, extracting durable knowledge from the ephemeral context.

Python 
from dataclasses import dataclass, field
from datetime import datetime
from typing import List, Dict, Optional
import hashlib

@dataclass
class MemoryRecord:
    """A single unit of long-term memory."""
    id: str
    content: str                    # The actual memory text
    memory_type: str                # "user", "feedback", "project", "reference"
    embedding: List[float]          # Vector for similarity search
    metadata: Dict[str, str]        # Structured tags for filtered retrieval
    created_at: datetime = field(default_factory=datetime.utcnow)
    last_accessed: datetime = field(default_factory=datetime.utcnow)
    access_count: int = 0
    importance_score: float = 0.5   # [0, 1] — learned from access patterns
    source_conversation_id: str = ""

    @property
    def dedup_key(self) -> str:
        return hashlib.sha256(self.content.encode()).hexdigest()[:16]


class LongTermMemory:
    """Persistent memory store with embedding-based retrieval."""

    def __init__(self, vector_store, embedder, consolidator_llm):
        self.vector_store = vector_store      # e.g., Pinecone, Qdrant, ChromaDB
        self.embedder = embedder              # Embedding model
        self.consolidator = consolidator_llm  # Small LLM for extraction

    async def consolidate(self, conversation: List[dict]) -> List[MemoryRecord]:
        """Extract durable memories from a completed conversation."""
        # Step 1: Ask the consolidator LLM to identify memorable information
        extraction_prompt = """Analyze this conversation and extract information
        worth remembering for future conversations. For each item, classify as:
        - user: facts about the user's identity, preferences, expertise
        - feedback: corrections or confirmations about how to work
        - project: decisions, goals, or context about the project
        - reference: pointers to external resources

        Only extract information that would be USEFUL in a future conversation
        where the same user returns. Skip ephemeral debugging details."""

        extracted = await self.consolidator.extract(
            extraction_prompt,
            conversation
        )

        # Step 2: Deduplicate against existing memories
        records = []
        for item in extracted:
            embedding = await self.embedder.embed(item["content"])

            # Check for near-duplicates (cosine similarity > 0.92)
            existing = await self.vector_store.query(
                vector=embedding,
                top_k=3,
                threshold=0.92
            )

            if existing:
                # Update existing memory rather than creating duplicate
                best_match = existing[0]
                best_match.access_count += 1
                best_match.last_accessed = datetime.utcnow()
                best_match.importance_score = min(
                    1.0,
                    best_match.importance_score + 0.1
                )
                await self.vector_store.update(best_match)
            else:
                record = MemoryRecord(
                    id=f"mem_{datetime.utcnow().timestamp()}",
                    content=item["content"],
                    memory_type=item["type"],
                    embedding=embedding,
                    metadata=item.get("metadata", {}),
                )
                records.append(record)
                await self.vector_store.upsert(record)

        return records

    async def retrieve(
        self,
        query: str,
        memory_types: Optional[List[str]] = None,
        top_k: int = 10,
        recency_weight: float = 0.3,
    ) -> List[MemoryRecord]:
        """Retrieve relevant memories with hybrid scoring."""
        query_embedding = await self.embedder.embed(query)

        # Vector similarity search with optional type filter
        candidates = await self.vector_store.query(
            vector=query_embedding,
            top_k=top_k * 3,  # Over-fetch for reranking
            filter={"memory_type": {"$in": memory_types}} if memory_types else None,
        )

        # Hybrid scoring: similarity × importance × recency
        now = datetime.utcnow()
        for record in candidates:
            age_days = (now - record.last_accessed).days
            recency = 1.0 / (1.0 + age_days * 0.05)  # Gentle decay

            record._score = (
                (1 - recency_weight) * record._similarity  # From vector search
                + recency_weight * recency
            ) * record.importance_score

        candidates.sort(key=lambda r: r._score, reverse=True)
        return candidates[:top_k]
Design Decision

Notice the importance score that increases every time a memory is accessed. This implements a form of spaced repetition — memories that the agent uses frequently become easier to retrieve over time. Memories that are stored but never accessed gradually sink in the rankings, effectively implementing forgetting without deletion.

The Cold Start Problem

A freshly deployed agent has no long-term memories. Every first interaction starts cold. There are three strategies for warming up LTM:

  1. Seed memories from documentation: Parse the project's README, architecture docs, and coding guidelines into memory records before the first conversation. This gives the agent baseline project knowledge.
  2. Import from prior systems: If migrating from a different agent or assistant, extract the previous conversation history and run the consolidation pipeline to pre-populate LTM.
  3. Aggressive early consolidation: In the first 5–10 conversations, lower the threshold for what gets consolidated. Accept more noise in exchange for faster memory bootstrapping. Tighten the threshold once the base is established.

5. Episodic Memory: Learning from Experience

Episodic memory stores specific events the agent experienced, tagged with context, actions taken, and outcomes. It's the difference between knowing "tests should be run before committing" (semantic knowledge) and remembering "last Tuesday, I committed without running tests on the payment service, the CI pipeline broke, and it took 45 minutes to fix because three other PRs had already merged on top" (episodic memory). The specificity is what makes it powerful — it captures not just rules but the consequences of violating them.

Cognitive Science Insight

In human cognition, episodic memories are distinguished by their autonoetic consciousness — the sense of "I was there, I did this, this happened." For agents, we approximate this by recording the agent's own actions and decisions (not just observations) along with the causal chain that followed. This enables the agent to learn from its own mistakes, not just from training data.

Episode Structure

A well-structured episode captures the full arc of an experience: the situation that triggered the action, the agent's reasoning, the action taken, and the outcome. This mirrors the SRAO pattern (Situation → Reasoning → Action → Outcome) from reinforcement learning.

Python 
from dataclasses import dataclass, field
from datetime import datetime
from enum import Enum
from typing import List, Dict, Optional

class Outcome(Enum):
    SUCCESS = "success"
    PARTIAL = "partial"
    FAILURE = "failure"
    UNKNOWN = "unknown"

@dataclass
class Episode:
    """A discrete experience the agent lived through."""
    id: str
    timestamp: datetime

    # Situation: What was the context?
    task_type: str                  # e.g., "code_review", "bug_fix", "refactor"
    situation: str                  # Natural language description
    environment: Dict[str, str]     # e.g., {"repo": "auth-service", "language": "python"}

    # Reasoning: What did the agent think?
    reasoning_trace: str            # The agent's chain of thought
    alternatives_considered: List[str] = field(default_factory=list)

    # Action: What did the agent do?
    actions_taken: List[str] = field(default_factory=list)
    tools_used: List[str] = field(default_factory=list)

    # Outcome: What happened?
    outcome: Outcome = Outcome.UNKNOWN
    outcome_details: str = ""
    error_message: Optional[str] = None
    resolution_time_seconds: Optional[int] = None

    # Learning: What should the agent take away?
    lesson_learned: Optional[str] = None  # Extracted by reflection
    embedding: List[float] = field(default_factory=list)


class EpisodicMemory:
    """Stores and retrieves experiential memories."""

    def __init__(self, vector_store, embedder):
        self.store = vector_store
        self.embedder = embedder

    async def record(self, episode: Episode) -> None:
        """Save an episode after a task completes."""
        # Embed the situation for similarity-based retrieval
        embed_text = f"{episode.task_type}: {episode.situation}"
        episode.embedding = await self.embedder.embed(embed_text)
        await self.store.upsert(episode)

    async def recall_similar(
        self,
        current_situation: str,
        task_type: Optional[str] = None,
        top_k: int = 5,
        outcome_filter: Optional[Outcome] = None,
    ) -> List[Episode]:
        """Find past episodes similar to the current situation."""
        query_embedding = await self.embedder.embed(current_situation)

        filters = {}
        if task_type:
            filters["task_type"] = task_type
        if outcome_filter:
            filters["outcome"] = outcome_filter.value

        episodes = await self.store.query(
            vector=query_embedding,
            top_k=top_k,
            filter=filters if filters else None,
        )
        return episodes

    async def recall_failures(
        self,
        current_situation: str,
        top_k: int = 3,
    ) -> List[Episode]:
        """Specifically retrieve past failures for preventive learning."""
        return await self.recall_similar(
            current_situation,
            outcome_filter=Outcome.FAILURE,
            top_k=top_k,
        )

    async def reflect(self, episode: Episode, llm) -> str:
        """Use an LLM to extract a lesson from a completed episode."""
        prompt = f"""Analyze this agent experience and extract a concise,
        actionable lesson that should guide future behavior in similar situations.

        Task type: {episode.task_type}
        Situation: {episode.situation}
        Actions taken: {', '.join(episode.actions_taken)}
        Outcome: {episode.outcome.value}
        Details: {episode.outcome_details}
        Error: {episode.error_message or 'None'}

        Lesson (1-2 sentences, specific and actionable):"""

        lesson = await llm.generate(prompt)
        episode.lesson_learned = lesson
        await self.store.update(episode)
        return lesson

Failure-Weighted Retrieval

The most valuable episodes are failures. A successful code review teaches the agent "this approach works" — which is useful but not surprising, since the agent presumably chose that approach because it seemed correct. A failed code review teaches "this approach seemed correct but wasn't" — that's the kind of information that prevents future mistakes.

In practice, this means episodic retrieval should be biased toward failures when the agent is about to take an action. Before committing code, the agent should ask: "Have I been in a similar situation before and failed? What went wrong?" This is the agent equivalent of the engineering post-mortem — except the agent actually reads its own post-mortems before every deployment.

Data Point

In internal experiments with agentic coding systems, agents with episodic memory that biased retrieval toward failures reduced repeated mistakes by 67% compared to agents using only semantic memory. The key mechanism: episodic memories carry emotional salience (success/failure tags) that raw knowledge lacks.

Episode Compression and Decay

Humans don't remember every episode at full fidelity forever. Details fade, but the gist — the lesson — persists. Agent episodic memory should follow the same pattern. After 30 days, detailed reasoning traces can be compressed to summaries. After 90 days, only the lesson learned and the outcome tag remain. This prevents the episodic store from growing unboundedly while preserving the most valuable signal.

6. Semantic Memory: The Knowledge Graph

Semantic memory stores facts, concepts, and relationships — knowledge that is independent of when or how it was learned. You know that Python uses indentation for scope, that HTTP 404 means "not found," and that PostgreSQL supports JSONB columns. You don't remember the specific moment you learned any of these things. That's semantic memory: knowledge stripped of its episodic context and distilled to pure understanding.

For an agentic system, semantic memory is the knowledge graph: a structured representation of entities, their properties, and the relationships between them. Where a vector store excels at fuzzy similarity search ("find things like this"), a knowledge graph excels at structured traversal ("given this function, what classes call it, what tests cover it, and what config files affect it?").

Implementation Note

The most effective production systems use both a vector store and a knowledge graph for semantic memory. The vector store handles natural language queries and fuzzy matching. The knowledge graph handles structured queries and relationship traversal. The Memory Router queries both and merges the results.

Building the Knowledge Graph

A semantic knowledge graph for a coding agent contains nodes (entities) and edges (relationships). The graph is built incrementally as the agent works — every file it reads, every function it calls, every error it encounters adds nodes and edges.

Python 
from dataclasses import dataclass, field
from datetime import datetime
from enum import Enum
from typing import List, Dict, Set, Optional, Tuple
from collections import defaultdict

class NodeType(Enum):
    FUNCTION = "function"
    CLASS = "class"
    MODULE = "module"
    API_ENDPOINT = "api_endpoint"
    CONFIG = "config"
    TEST = "test"
    CONCEPT = "concept"       # Abstract concepts: "authentication", "caching"
    PERSON = "person"         # Team members, code owners
    SERVICE = "service"       # Microservice boundaries

class EdgeType(Enum):
    CALLS = "calls"
    INHERITS = "inherits"
    IMPORTS = "imports"
    TESTS = "tests"
    CONFIGURES = "configures"
    OWNS = "owns"             # Person → Module
    DEPENDS_ON = "depends_on" # Service → Service
    RELATES_TO = "relates_to" # Concept → Concept
    IMPLEMENTS = "implements" # Function → Concept

@dataclass
class KnowledgeNode:
    id: str
    name: str
    node_type: NodeType
    properties: Dict[str, str] = field(default_factory=dict)
    embedding: List[float] = field(default_factory=list)
    last_updated: datetime = field(default_factory=datetime.utcnow)
    confidence: float = 1.0  # How sure are we this is still accurate?

@dataclass
class KnowledgeEdge:
    source_id: str
    target_id: str
    edge_type: EdgeType
    weight: float = 1.0      # Strength of relationship
    evidence: str = ""       # Why we believe this edge exists
    last_verified: datetime = field(default_factory=datetime.utcnow)


class SemanticMemory:
    """Knowledge graph with vector-augmented retrieval."""

    def __init__(self, graph_db, vector_store, embedder):
        self.graph = graph_db       # e.g., Neo4j, Amazon Neptune
        self.vectors = vector_store # Parallel vector index for fuzzy search
        self.embedder = embedder
        self._adjacency: Dict[str, Set[Tuple[str, EdgeType]]] = defaultdict(set)

    async def add_node(self, node: KnowledgeNode) -> None:
        if not node.embedding:
            node.embedding = await self.embedder.embed(
                f"{node.node_type.value}: {node.name} — "
                + " ".join(f"{k}={v}" for k, v in node.properties.items())
            )
        await self.graph.upsert_node(node)
        await self.vectors.upsert(node.id, node.embedding, {
            "name": node.name,
            "type": node.node_type.value,
        })

    async def add_edge(self, edge: KnowledgeEdge) -> None:
        await self.graph.upsert_edge(edge)
        self._adjacency[edge.source_id].add((edge.target_id, edge.edge_type))
        self._adjacency[edge.target_id].add((edge.source_id, edge.edge_type))

    async def query_context(
        self,
        entity_name: str,
        depth: int = 2,
        max_nodes: int = 50,
    ) -> Dict:
        """Retrieve a subgraph around an entity for context injection.

        This is the core retrieval operation for semantic memory:
        given something the agent is working on, pull the local
        neighborhood of related knowledge.
        """
        # Step 1: Find the entity (exact match or fuzzy via embedding)
        root = await self.graph.find_node(name=entity_name)
        if not root:
            embedding = await self.embedder.embed(entity_name)
            candidates = await self.vectors.query(embedding, top_k=3)
            if not candidates:
                return {"nodes": [], "edges": []}
            root = await self.graph.get_node(candidates[0].id)

        # Step 2: BFS to collect the local subgraph
        visited = {root.id}
        frontier = [root.id]
        collected_nodes = [root]
        collected_edges = []

        for _ in range(depth):
            next_frontier = []
            for node_id in frontier:
                neighbors = await self.graph.get_neighbors(node_id)
                for neighbor, edge in neighbors:
                    collected_edges.append(edge)
                    if neighbor.id not in visited and len(collected_nodes) < max_nodes:
                        visited.add(neighbor.id)
                        collected_nodes.append(neighbor)
                        next_frontier.append(neighbor.id)
            frontier = next_frontier

        # Step 3: Format as injectable context
        return {
            "nodes": [
                {"name": n.name, "type": n.node_type.value, **n.properties}
                for n in collected_nodes
            ],
            "edges": [
                {"from": e.source_id, "to": e.target_id, "rel": e.edge_type.value}
                for e in collected_edges
            ],
            "summary": self._generate_summary(collected_nodes, collected_edges),
        }

    def _generate_summary(self, nodes, edges) -> str:
        """Create a natural-language summary of the subgraph."""
        lines = []
        by_type = defaultdict(list)
        for n in nodes:
            by_type[n.node_type.value].append(n.name)

        for ntype, names in by_type.items():
            lines.append(f"  {ntype}: {', '.join(names[:10])}")

        edge_summary = defaultdict(int)
        for e in edges:
            edge_summary[e.edge_type.value] += 1

        rel_parts = [f"{count} {rel}" for rel, count in edge_summary.items()]
        lines.append(f"  relationships: {', '.join(rel_parts)}")

        return "Knowledge context:\n" + "\n".join(lines)

Staleness and Confidence Decay

Knowledge graphs rot. A function gets renamed, a service gets decomposed, an API endpoint changes its contract. If the knowledge graph doesn't track when each node was last verified, the agent will confidently assert facts that are no longer true — a particularly insidious form of hallucination because the information was true when stored.

The confidence field on each node implements a decay function: nodes that haven't been verified in 30+ days drop to 0.8 confidence, 90+ days to 0.5, 180+ days to 0.3. When the agent retrieves a low-confidence node, it can either verify it before using it (by reading the actual file) or caveat its response with the staleness. This is the knowledge graph equivalent of checking the date on a Wikipedia article.

Days Since Verification Confidence Agent Behavior
0–7 1.0 Use directly — information is fresh
8–30 0.9 Use with minor caveat, no verification needed
31–90 0.7 Verify before acting on it — read the actual source
91–180 0.4 Treat as a hint, always verify, flag to user
180+ 0.2 Consider stale — retrieve but don't rely on

7. Procedural Memory: Learning How to Act

Procedural memory stores learned sequences of actions — the "how to" knowledge. You don't consciously think about how to ride a bicycle; your procedural memory executes a complex motor sequence automatically. Similarly, a mature agent shouldn't re-derive its debugging workflow from first principles every time a test fails. It should have a stored procedure: (1) read the error message, (2) find the failing test, (3) read the relevant source code, (4) check recent git changes to that file, (5) form a hypothesis, (6) verify.

Procedural memory is the most under-explored memory type in current agentic systems — and arguably the most impactful for efficiency. Every time an agent re-discovers a workflow it has already executed successfully, it wastes tokens, time, and user patience.

"A senior engineer doesn't think about how to debug — they just debug. Procedural memory is what separates an agent that reasons about debugging from an agent that simply debugs."

Procedure Structure

A procedure is a reusable workflow template with three key properties: a trigger condition (when to activate it), an action sequence (what steps to take), and a success metric (how to know if it worked). Procedures evolve through execution — successful runs reinforce the procedure, failed runs trigger modification.

from datetime import datetime

from enum import Enum

class StepType(Enum):
    TOOL_CALL = "tool_call"       # Invoke a specific tool
    LLM_REASON = "llm_reason"    # Ask the LLM to analyze/decide
    CONDITIONAL = "conditional"   # Branch based on previous output
    LOOP = "loop"                # Repeat until condition met
    HUMAN_CHECK = "human_check"  # Pause for human review

@dataclass
class ProcedureStep:
    """A single step in a learned procedure."""
    order: int
    step_type: StepType
    description: str              # Human-readable description
    tool_name: Optional[str] = None  # For TOOL_CALL steps
    tool_args_template: Optional[Dict] = None  # Parameterized args
    condition: Optional[str] = None  # For CONDITIONAL/LOOP steps
    fallback_step: Optional[int] = None  # Jump here on failure

@dataclass
class Procedure:
    """A learned, reusable workflow."""
    id: str
    name: str                     # e.g., "debug_failing_test"
    trigger_pattern: str          # Natural language: "when a test fails"
    trigger_embedding: List[float] = field(default_factory=list)

    steps: List[ProcedureStep] = field(default_factory=list)
    parameters: List[str] = field(default_factory=list)  # Input params

    # Performance tracking
    execution_count: int = 0
    success_count: int = 0
    avg_duration_seconds: float = 0.0
    last_executed: Optional[datetime] = None

    # Evolution
    version: int = 1
    parent_id: Optional[str] = None  # Forked from this procedure
    created_from_episode: Optional[str] = None  # Which episode inspired it

    @property
    def success_rate(self) -> float:
        if self.execution_count == 0:
            return 0.0
        return self.success_count / self.execution_count

    @property
    def reliability_score(self) -> float:
        """Bayesian-adjusted score: more executions = more confidence."""
        # Wilson score interval lower bound
        if self.execution_count == 0:
            return 0.0
        n = self.execution_count
        p = self.success_rate
        z = 1.96  # 95% confidence
        denominator = 1 + z**2 / n
        center = p + z**2 / (2 * n)
        spread = z * (p * (1 - p) / n + z**2 / (4 * n**2)) ** 0.5
        return (center - spread) / denominator


class ProceduralMemory:
    """Stores and retrieves learned action sequences."""

    def __init__(self, vector_store, embedder):
        self.store = vector_store
        self.embedder = embedder
        self.procedures: Dict[str, Procedure] = {}

    async def learn_from_episode(self, episode, llm) -> Optional[Procedure]:
        """Extract a reusable procedure from a successful episode."""
        if episode.outcome.value != "success":
            return None

        # Ask the LLM to generalize the episode into a procedure
        prompt = f"""This agent successfully completed a task. Extract a
        reusable procedure that could be applied to similar future tasks.

        Task type: {episode.task_type}
        Situation: {episode.situation}
        Actions taken (in order):
        {chr(10).join(f'  {i+1}. {a}' for i, a in enumerate(episode.actions_taken))}
        Tools used: {', '.join(episode.tools_used)}

        Create a generalized procedure with:
        1. A trigger pattern (when should this procedure activate?)
        2. Parameterized steps (replace specific values with parameters)
        3. Conditional branches (what if a step fails?)"""

        procedure_spec = await llm.generate(prompt)
        procedure = self._parse_procedure(procedure_spec, episode)

        if procedure:
            procedure.trigger_embedding = await self.embedder.embed(
                procedure.trigger_pattern
            )
            self.procedures[procedure.id] = procedure
            await self.store.upsert(procedure)

        return procedure

    async def match(self, situation: str, top_k: int = 3) -> List[Procedure]:
        """Find procedures that match the current situation."""
        embedding = await self.embedder.embed(situation)
        candidates = await self.store.query(embedding, top_k=top_k * 2)

        # Filter by reliability and sort by composite score
        viable = [
            p for p in candidates
            if p.execution_count < 3 or p.success_rate >= 0.5
        ]

        viable.sort(
            key=lambda p: (
                p.reliability_score * 0.6
                + p._similarity * 0.4  # From vector search
            ),
            reverse=True,
        )
        return viable[:top_k]

    async def record_execution(
        self, procedure_id: str, success: bool, duration_seconds: float
    ) -> None:
        """Update a procedure's performance stats after execution."""
        proc = self.procedures.get(procedure_id)
        if not proc:
            return

        proc.execution_count += 1
        if success:
            proc.success_count += 1
        proc.last_executed = datetime.utcnow()

        # Running average of duration
        proc.avg_duration_seconds = (
            proc.avg_duration_seconds * (proc.execution_count - 1)
            + duration_seconds
        ) / proc.execution_count

        await self.store.update(proc)

        # If success rate drops below threshold, flag for review
        if proc.execution_count >= 5 and proc.success_rate < 0.4:
            proc.name = f"[NEEDS REVIEW] {proc.name}"
            await self.store.update(proc)

The Overfitting Trap

Procedures extracted from a single episode are dangerously specific. A debugging procedure learned from a React component failure might include "check the useEffect dependency array" — which is useless for a Python debugging task. Production systems should require at least 3 successful episodes of the same task type before extracting a procedure, and the extraction prompt should explicitly ask the LLM to abstract away domain-specific details.

Procedure Evolution

Procedures shouldn't be static. When a procedure fails, the system should analyze why — was it the wrong trigger match, a missing step, or a changed environment? Based on the failure mode, the procedure is either patched (add a missing step), branched (create a variant for a different context), or deprecated (mark as unreliable and stop matching it).

This creates a Darwinian selection pressure on procedures: effective ones get reinforced and rise in the matching rankings; ineffective ones get modified or die out. Over time, the procedural memory converges on a set of battle-tested workflows that the agent can execute with high confidence and minimal wasted reasoning.

8. The Memory Router: Orchestrating Retrieval

Five memory systems, one context window. The Memory Router is the critical component that decides which memories to retrieve from which systems, and how much context window budget to allocate to each. Without it, each memory system competes for space and the agent suffers from context pollution — too many irrelevant memories displacing the information it actually needs.

Memory System Default Budget Retrieval Trigger Scaling Behavior
Short-Term 40% of context Always present (conversation history) Grows with conversation length, evicts when full
Long-Term 15% of context New session, new user, topic shift More memories retrieved for ambiguous queries
Episodic 15% of context Before risky actions, after errors More episodes recalled for unfamiliar task types
Semantic 20% of context Entity mentioned, code context needed Deeper graph traversal for complex dependencies
Procedural 10% of context Task type identified, action planning Single procedure loaded (steps are compact)

These budgets are adaptive, not fixed. When the agent is debugging a complex issue, semantic memory might need 40% of the budget to hold the dependency graph, while procedural memory needs only 5% (a single debugging procedure). When the agent is doing routine code formatting, procedural memory dominates and semantic context is minimal. The router adjusts based on task type and retrieval signal strength.

Key Architectural Insight

The Memory Router performs parallel retrieval, then sequential ranking. It fires queries to all five memory systems simultaneously (latency = max of individual systems, not sum), collects candidates, applies a cross-system relevance ranking, and fills the context window from highest to lowest relevance until the budget is exhausted. This is the attention mechanism — but at the memory system level rather than the token level.

9. Tradeoffs and Design Decisions

Building a five-system memory architecture involves sharp engineering tradeoffs. Here are the decisions that will define your system's behavior:

Decision Option A Option B Recommendation
Embedding model Same model as the agent LLM (unified embedding space) Dedicated embedding model (e.g., text-embedding-3-large) Dedicated model — 10x cheaper, often better retrieval quality for search
Consolidation timing Real-time (extract memories during conversation) Post-session (batch extraction after conversation ends) Post-session — avoids latency spikes during active work, allows full-context extraction
Knowledge graph storage Dedicated graph DB (Neo4j, Neptune) Property graph in PostgreSQL (with ltree/jsonb) PostgreSQL for <100K nodes — simpler ops, good enough traversal. Neo4j for larger graphs
Procedure learning Automatic extraction from episodes Human-authored procedure templates Start with human-authored seed procedures, enable automatic extraction after baseline is established
Memory isolation Shared memory across all users Isolated memory per user/project Isolated by default — shared only for organizational knowledge (e.g., coding standards)
Eviction strategy Hard deletion after TTL expires Soft decay (lower confidence/importance over time) Soft decay — hard deletion risks losing valuable rare memories. Let retrieval ranking handle relevance
The Biggest Mistake

The most common failure mode in agentic memory systems is retrieval pollution — surfacing so many memories that the context window is full of retrieved context and has no room for the actual conversation. A good heuristic: retrieved memories should never exceed 40% of the total context window. If retrieval is consuming more than that, your relevance ranking is too permissive.

10. Production Considerations

Cost Model

Memory systems add cost at three points: embedding generation (write path), vector search (read path), and context window consumption (inference path). Here's how the costs break down for a moderately active agent handling ~100 conversations per day:

Component Operation Unit Cost Daily Cost (100 conv)
Embedding generation Consolidation writes + retrieval queries $0.0001 / 1K tokens ~$0.50
Vector store Storage + queries (Pinecone/Qdrant) $0.08 / 1M reads ~$0.10
Knowledge graph Node storage + traversal queries Varies (self-hosted ≈ $50/mo) ~$1.60
LLM consolidation Post-session memory extraction $0.01–0.05 per conversation ~$3.00
Context window Additional tokens from retrieved memories $0.003 / 1K input tokens ~$9.00
Total daily incremental cost for memory ~$14.20

The dominant cost is context window consumption — the memories you retrieve cost money every time they're included in the prompt. This is why aggressive relevance ranking matters: every irrelevant memory you don't include saves both money and attention budget.

Privacy and Data Governance

Agentic memory raises thorny privacy questions that traditional databases don't. When a user tells an agent "I'm going through a divorce and need to restructure my codebase accordingly," should that personal information be stored? If so, for how long? Who can access it?

  • Memory opt-out: Users must be able to see what the agent remembers about them and delete specific memories. This is analogous to GDPR's right to erasure.
  • Content classification: The consolidation pipeline should classify extracted memories as "professional" or "personal" and apply different retention policies to each.
  • Access boundaries: Memories stored in a work context should never leak into personal contexts, and vice versa. Namespace isolation by project/organization is the minimum.
  • Audit trails: Every memory write should be logged with its source conversation, so you can trace back why the agent "knows" something and verify it wasn't poisoned by a prompt injection attack.

11. Conclusion: Memory Makes the Agent

We started with a provocation: an LLM without memory is a brilliant amnesiac. We end with an architecture: five memory systems — short-term, long-term, episodic, semantic, and procedural — orchestrated by a Memory Router that packs the right memories into a finite context window at the right time.

This isn't just infrastructure. It's the difference between an agent that solves the same problem the same way every time and an agent that gets better. Episodic memory prevents repeated mistakes. Semantic memory provides deep context without expensive re-reading. Procedural memory automates learned workflows. Long-term memory maintains continuity across sessions. And short-term memory ensures the most relevant information is always in focus.

"The question is no longer 'can the model reason?' It can. The question is 'can the model remember?' — and the answer depends entirely on the memory architecture you build around it."

Every agent system in this series — the code agent, the SOC agent, the drug discovery agent, the supply chain network — would be fundamentally improved by implementing the five-system memory architecture described here. Memory is not a feature. It's the foundation. Build it first, and everything else gets easier.
