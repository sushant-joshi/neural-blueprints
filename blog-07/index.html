Inside Claude Code: From Keystroke to Response

You type a prompt. Seconds later, Claude is reading your files, editing your code, running your tests, and explaining what it changed. But what actually happens in between? This essay traces the complete lifecycle of a Claude Code request — from terminal input through the agentic loop, tool execution, MCP integration, and the massive GPU clusters that power inference — with the technical depth to understand every layer.

SJ
Sushant Joshi

Claude Code is a ~512,000-line TypeScript application. It runs on the Bun runtime, renders its terminal UI with React and Ink, and orchestrates an agentic loop that can chain dozens of tool calls in a single conversation. But here's the number that should stop you: only 1.6% of those 512,000 lines are AI decision logic. The other 98.4% is deterministic infrastructure — permission gates, context management, tool routing, sandbox enforcement, and recovery logic. The intelligence is in the model. The engineering is in everything around it.

1. The Request Lifecycle: What Happens When You Hit Enter

When you type a prompt and press Enter in Claude Code, the request passes through six distinct phases before you see a response. Understanding these phases is key to understanding why Claude Code feels fast for some operations and slow for others — and where the engineering complexity actually lives.

Phase 1: Context Assembly

Before your prompt is ever sent to an LLM, Claude Code assembles a context payload from multiple sources. This happens locally, on your machine, in milliseconds:

  • System prompt (~4,200 tokens) — The behavioral instructions that define how Claude Code operates: tool usage patterns, safety constraints, output formatting rules. This is always loaded and hidden from you.
  • CLAUDE.md files (variable) — Project-specific instructions from your repository root, nested directories, and user-level config. These are injected as system context so Claude understands your project's conventions.
  • Auto-memory (~680 tokens) — The first 200 lines (or 25KB) of your MEMORY.md file, containing information Claude saved from previous sessions about you, your project, and your preferences.
  • Environment info (~280 tokens) — Your working directory, platform (macOS/Linux/Windows), shell, git state, OS version, and current model ID.
  • Tool definitions (~120 tokens) — Names of all available tools (built-in + MCP). Full schemas are deferred by default — only loaded when Claude actually uses a tool.
  • Conversation history — All previous messages in the current session, including tool calls and their results.
Eye-Opening Insight

Tool schema deferral is a critical optimization. With 50+ tools, loading all schemas upfront would consume thousands of tokens. Instead, only tool names are loaded (~120 tokens total), and Claude uses a ToolSearch meta-tool to fetch full schemas on demand. This keeps the context window lean while giving Claude access to an unlimited number of tools.

Phase 2: API Submission

The assembled context is sent to the Claude API as a standard Messages API request over HTTPS. The request includes the compiled message array, tool definitions, model selection, and streaming configuration. Claude Code supports multiple API backends:

Phase 3: Streaming Response

The API returns a response stream using Server-Sent Events (SSE). This is the mechanism behind the "typing" effect you see in your terminal. The event flow follows a precise sequence:

  1. message_start — Initial metadata and token usage counters
  2. content_block_start — Signals the beginning of a content block (text or tool call)
  3. content_block_delta — Individual token deltas arrive as the model generates them. For text, this is character-by-character. For tool calls, it's the JSON payload being assembled incrementally.
  4. content_block_stop — End of a single content block
  5. message_delta — Final metadata with total output token count
  6. message_stop — Stream complete

The terminal UI (built with React + Ink) renders these deltas in real-time. Text chunks appear as they arrive; tool call blocks are buffered until complete, then displayed as formatted tool invocations.

Phases 4–6: The Agentic Loop

If the response contains only text (stop reason: end_turn), the cycle is complete. But if the response contains tool call blocks, phases 4–6 kick in — and this is where Claude Code becomes an agent rather than a chatbot. We'll cover this in detail next.

2. The Agentic Loop: Where Intelligence Meets Infrastructure

The heart of Claude Code is a while(true) loop. It's roughly 1,700 lines of TypeScript in the core query engine, and its logic is deceptively simple:

  1. Send messages to Claude API with tool definitions
  2. Receive response (streamed)
  3. If response contains text → display to user
  4. If response contains tool calls → execute each tool, capture results
  5. Append tool results as new messages in the conversation
  6. Go to step 1
  7. Loop terminates when Claude returns stop_reason: "end_turn" with no tool calls

"The agentic loop is architecturally trivial — send, receive, execute, repeat. The engineering complexity isn't in the loop. It's in everything the loop touches: permissions, sandboxing, context management, error recovery, and tool routing."

What makes this powerful is that Claude controls the loop. There's no hard-coded planner, no state machine, no pre-defined workflow. Claude decides which tool to call, what arguments to pass, how to interpret the result, and when to stop. The infrastructure's job is to make those decisions safe and fast.

Example: "Fix the login bug"

→ Bash(git log --oneline -10) — Claude checks recent changes

→ Grep("login" in src/**/*.ts) — Finds login-related files

→ Read(src/auth/session.ts) — Examines the authentication code

→ Read(src/auth/middleware.ts) — Reads related middleware

→ Edit(src/auth/session.ts, lines 45-52) — Makes the fix

→ Bash(npm test -- --grep "session") — Runs relevant tests

→ end_turn — "Fixed. The session validation was checking expiry against UTC instead of local time."

That's 6 tool calls, 7 API round-trips, all orchestrated autonomously by the model.

Subagents: Parallel Agentic Loops

Claude Code can spawn subagents — isolated instances of the agentic loop that run in parallel. Each subagent has its own conversation context, its own tool permissions, and its own termination condition. The parent agent delegates a task ("search the codebase for all API endpoints"), the subagent executes it independently, and returns a single result message.

Subagents serve two purposes: parallelism (multiple searches or tasks running concurrently) and context isolation (preventing large tool outputs from polluting the parent's context window). A subagent can read 50 files and return a 200-word summary, rather than the parent consuming 50,000 tokens of raw file content.

3. Tool Execution: The Permission-Gated Pipeline

Claude Code ships with 40+ built-in tools spanning file operations (Read, Edit, Write, Glob, Grep), execution (Bash), orchestration (Agent, Skill), web access (WebFetch, WebSearch), and more. Every tool call passes through a multi-stage pipeline before execution.

The Permission Evaluation Chain

When Claude generates a tool call, it doesn't execute immediately. The tool call enters a three-layer permission evaluation:

  1. Pre-hook check: If you've configured hooks, a shell script runs first and can block the call outright by returning exit code 2. Hooks receive the full tool call as JSON on stdin.
  2. Permission rules evaluation: Rules are checked in strict priority order — deny rules first (always block), then ask rules (prompt the user), then allow rules (auto-execute). Rules use glob-style matching: Bash(npm run *) allows any npm run command; Edit(/etc/*) blocks editing system files.
  3. User prompt: If no rule matches, the default behavior is to prompt the user. The terminal displays the tool call and waits for approval.

Only after passing all three layers does the tool actually execute. After execution, a post-hook can run — useful for logging, notifications, or formatting results before they enter the conversation.

Defense in Depth

Permissions and sandboxing are separate layers. Permissions decide whether a tool runs. Sandboxing constrains what it can do once running. A Bash command might be allowed by permission rules but still filesystem-restricted by the sandbox. On macOS, sandboxing uses Seatbelt (Apple's kernel-level sandbox); on Linux, it uses bubblewrap. This means even if a permission rule is too broad, the sandbox provides a hard OS-level enforcement boundary.

Sandbox Enforcement

The sandbox provides filesystem and network isolation for Bash commands:

  • Filesystem: By default, commands can read the entire machine but only write to the working directory and its subdirectories. Write access to additional paths (like ~/.kube or /tmp/build) must be explicitly granted.
  • Network: Commands route through a proxy that enforces a domain allowlist. Only approved domains are reachable — this covers curl, wget, npm install, and every other network-using tool.
  • Inheritance: All child processes inherit the sandbox boundaries. A Bash command can't escape the sandbox by spawning a subprocess.

4. MCP: The Universal Tool Protocol

The Model Context Protocol (MCP) is Anthropic's open standard for connecting AI to external tools and data sources. Introduced in November 2024 and now adopted by OpenAI, Google DeepMind, and others, MCP solves the N x M integration problem: instead of every model needing custom integrations with every tool, both sides implement one standard protocol.

How MCP Servers Integrate with Claude Code

An MCP server is a subprocess that Claude Code spawns at session start. It communicates over JSON-RPC 2.0 via stdio (standard input/output) or SSE (Server-Sent Events). The lifecycle:

  1. Startup: Claude Code reads MCP server configs from .claude/settings.json and spawns each server as a child process with the specified command and arguments.
  2. Tool discovery: Claude Code sends a ListTools JSON-RPC call. The server responds with the name, description, and JSON Schema for each tool it provides.
  3. Deferred loading: By default, only tool names are injected into the context. Full schemas are loaded on demand via the ToolSearch meta-tool when Claude decides to use a specific tool.
  4. Invocation: When Claude calls an MCP tool, Claude Code sends a CallTool JSON-RPC request to the appropriate server with the tool name and arguments.
  5. Result: The server executes the operation and returns the result. This result is formatted as a tool response message and fed back into the agentic loop.

MCP tools follow the same permission model as built-in tools. You can allow mcp__github__* (all GitHub MCP tools) or deny mcp__github__create_secret (block a specific dangerous operation) using the same glob-style rules.

Why MCP Matters

Before MCP, every AI tool integration was bespoke — a custom function, a custom schema, a custom auth flow. MCP standardizes this into a single protocol with three primitives: Tools (actions the AI can take), Resources (data the AI can read), and Prompts (templates the AI can use). Any MCP server works with any MCP client, from Claude Code to VS Code to custom agents. Build the server once, use it everywhere.

5. Skills: Reusable Prompt Workflows

Skills are Markdown files with YAML frontmatter that encode reusable workflows. When invoked (via /skill-name or automatically by Claude), the skill's content is injected into the conversation as a detailed instruction set.

The key architectural insight: skills are not code. They're prompts. A skill doesn't execute procedural logic — it provides Claude with domain-specific instructions that shape how it uses the existing tool infrastructure. A /commit skill doesn't call git commit directly; it tells Claude how to create good commits (check status, read diff, write a message following conventions, stage specific files, etc.) and Claude executes that workflow using the Bash tool.

Skills support parameter substitution ($ARGUMENTS, $0, $1), can be restricted to specific tools via allowed-tools in frontmatter, and can run in isolated subagent contexts with context: fork. Skill descriptions load at session start (consuming roughly 1% of context budget), and full content loads only on invocation.

6. Context Window Management: The Hidden Bottleneck

The context window is simultaneously Claude Code's most powerful feature and its hardest engineering constraint. Modern Claude models support up to 1 million tokens of context — but that budget fills fast when every file read, every Bash output, and every tool result consumes tokens.

What Consumes the Budget

ComponentToken BudgetBehavior
System prompt~4,200 tokensAlways loaded, never evicted
CLAUDE.md filesVariableLoaded per session, re-attached after compaction
Auto-memory (MEMORY.md)~680 tokens (25KB / 200 lines max)Loaded at session start
Environment info~280 tokensAlways loaded
MCP tool names~120 tokens (for 50+ tools)Names only; schemas deferred
Skill descriptions~1% of context windowLoaded at start; full content on invocation
Conversation historyAll remaining capacityGrows continuously; compacted when full

Three-Timescale Compaction

When the context window approaches capacity, Claude Code triggers automatic compaction through a three-layer strategy:

  1. Short-term (per-message): The oldest tool outputs are cleared first. A Bash command that returned 5,000 tokens of npm install output three turns ago is expendable — the decision it informed has already been made.
  2. Mid-term (session): If clearing tool outputs isn't enough, the conversation itself is summarized. Claude Code generates a compressed summary of the conversation so far, preserving key decisions, code snippets, and user instructions while discarding the verbose back-and-forth.
  3. Long-term (multi-session): Important information that should persist across sessions is written to auto-memory (MEMORY.md). This includes user preferences, project context, and feedback — the things that make Claude Code feel like it "knows" your project.
Practical Limit

While the theoretical context window is up to 1M tokens, Claude Code's practical productive limit is lower. Performance degrades as context fills — retrieval accuracy drops, reasoning becomes less focused, and latency increases. The recommendation is to run /compact at ~60% utilization, well before quality degrades. Think of it like RAM: your computer has 16GB, but you don't want every byte occupied.

7. The Infrastructure: What Happens on the Server Side

When your request leaves Claude Code and hits the API endpoint, it enters one of the largest AI compute infrastructures in the world. Anthropic runs a multi-cloud, multi-chip strategy that spans three hyperscalers and custom data center builds.

Different chips excel at different workloads: Trainium2 is optimized for training, TPUv7 delivers superior price-per-token for inference, and NVIDIA GPUs provide flexibility. By running across all three, Anthropic can route workloads to the most efficient hardware for each task.

Prompt Caching: The Latency Cheat Code

Claude Code benefits enormously from prompt caching. The system prompt, CLAUDE.md files, and tool definitions are identical across turns in a session — they're cached server-side after the first request. With an 80% cache hit rate, a request with a 2M input token limit effectively gets 10M tokens of throughput. More importantly, cached tokens reduce latency by up to 85% for long prompts. This is why Claude Code's second response in a session is noticeably faster than the first.

8. The Agentic Loop in Practice: Latency Breakdown

Understanding where time is spent in the agentic loop explains why some Claude Code interactions feel instant and others take minutes:

PhaseTypical LatencyWhat Determines It
Context assembly (local)5–50msNumber of CLAUDE.md files, MCP servers to query
Network round-trip50–200msGeographic distance to API endpoint
Time to first token0.6–1.1sModel size (Haiku vs Opus), prompt length

Tool execution (Bash)100ms–60s+Command complexity (ls vs npm install)
Permission prompt (user)0–∞Whether user needs to approve the tool call

A simple question-and-answer interaction requires one API round-trip: ~2 seconds. A complex bug fix with 6 tool calls requires 7 round-trips plus 6 tool executions: potentially 15–30 seconds of model time plus tool execution time. The dominant factor is usually the number of agentic loop iterations, not any single step.

Why Pre-Approved Permissions Matter

In the latency breakdown above, the user permission prompt is the only step with unbounded latency. If Claude needs to ask for permission on every tool call, a 6-tool-call task requires 6 interruptions. This is why configuring allow rules for common operations (Bash(npm run *), Bash(git *), Read(./**)) dramatically improves the agentic loop's throughput — Claude can chain tool calls without pausing for approval.

9. Eye-Opening Insights: What Most People Get Wrong

Insight 1: The 98.4% Rule

The most counterintuitive fact about Claude Code is that almost none of it is "AI." The model provides reasoning and decision-making through the API — that's the 1.6%. The other 512,000 lines handle: terminal rendering, permission evaluation, tool routing, sandbox enforcement, context management, session persistence, MCP communication, hook execution, error recovery, and crash protection. Building an agentic coding tool is not primarily an AI problem. It's a systems engineering problem.

Insight 2: The Model Controls the Loop

There's no planner. No state machine. No workflow engine. Claude Code gives the model a set of tools and asks "what do you want to do?" The model responds with a tool call (or text). The infrastructure executes it. The result goes back. The model decides again. This is radical because it means Claude Code's capabilities scale automatically with model improvements — a smarter model makes better tool-use decisions without any infrastructure changes.

"Claude Code doesn't orchestrate Claude. It creates an environment where Claude can orchestrate itself — safely."

Insight 3: Sessions Are Just JSONL Files

Every Claude Code session is persisted as a JSONL (JSON Lines) file in ~/.claude/projects/. Each line is a message — your prompts, Claude's responses, tool calls, tool results. This means sessions are resumable, forkable, inspectable, and even editable. If Claude Code crashes mid-task, it resumes from the last checkpoint. If you want to understand what Claude did, you can read the session file. There's no opaque state — the entire conversation is a transparent, append-only log.

Insight 4: The Source Map Leak

In March 2026, an npm packaging error shipped Claude Code v2.1.88 with a 59.8MB source map file (cli.js.map) that allowed full reconstruction of the TypeScript source. This accidental disclosure revealed the 512K-line codebase to the community, leading to independent rewrites in Rust (~20,000 lines) and Python, and validated that the architecture was indeed a straightforward agentic loop rather than some exotic proprietary framework.

Insight 5: Prompt Caching Makes Everything Viable

Without prompt caching, Claude Code would be economically impractical. The system prompt (4,200 tokens) plus tool definitions and CLAUDE.md files might total 10,000+ tokens, sent on every single API call in the agentic loop. A 6-tool-call task would bill 60,000+ input tokens just for repeated context. With caching, those tokens are sent once and referenced by cache key on subsequent calls — reducing per-turn cost by up to 90% and latency by up to 85%. Prompt caching isn't an optimization; it's an architectural prerequisite for the agentic loop to be viable at scale.

The elegance of this architecture is that the model sees the same interface regardless of complexity. A one-shot question and a 50-tool-call refactoring job use the same loop, the same permission system, the same context management. Scale comes from the model's capabilities, not from the infrastructure's complexity. And that's the deepest insight of all: the best agentic architecture is the one that gets out of the model's way.
