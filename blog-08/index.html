Skills, Rules, Hooks, MCPs, Plugins: The Same Problem, Five Different Answers

You want your AI coding agent to work with GitHub — create PRs, enforce conventions, check CI status. Do you write a Rule? A Skill? A Hook? An MCP server? A Plugin? The answer is "all of them, for different reasons." This essay uses one concrete example to cut through the terminology fog and reveal why these mechanisms exist, how they differ, and when each one is the right choice.

Sushant Joshi
Principal Engineer · AI Systems
· April 27, 2026  ·  26 min read

Every AI coding agent — Claude Code, GitHub Copilot, Cursor, Windsurf, OpenAI Codex — has invented its own vocabulary for extending the agent's capabilities. Rules. Skills. Hooks. MCPs. Plugins. Actions. Custom Instructions. Gems. Extensions. The terminology is a mess, and under the surface, most of these concepts are solving variations of the same five problems. This essay maps the territory by taking a single, concrete integration — GitHub — and implementing it five different ways. Not because you should, but because each mechanism handles a fundamentally different aspect of the problem, and understanding which aspect reveals the real architecture beneath the labels.

"The confusion isn't that these terms mean different things. It's that they solve different problems that feel like the same problem until you try to implement them."

1. The Example: "Make My AI Agent Work With GitHub"

Here's what most engineers actually want when they say "GitHub integration":

  1. Know our conventions — commit message format, branch naming, PR template, required reviewers
  2. Create PRs properly — with the right title format, body template, labels, and linked issues
  3. Never force-push to main — a hard safety constraint, no exceptions
  4. Read issues, check CI status, post comments — interact with GitHub's API programmatically
  5. Auto-label PRs by file path — when a PR touches src/auth/*, add the "security" label

Five requirements. It's tempting to think one mechanism should handle all of them. But each requirement has different characteristics — different answers to questions like: Does it need to be deterministic? Does it need API access? Does it run before or after the AI acts? Is it a suggestion or a hard constraint? Those differences determine which mechanism fits.

Let's implement each requirement with the mechanism that fits it best, and understand why it fits.

2. Rules: Persistent Behavioral Constraints

Requirement #1: Know our conventions

"Commit messages must follow Conventional Commits format. Branch names must be feature/JIRA-123-description or fix/JIRA-456-description. PRs must link to an issue. Never commit .env files."

A Rule is a piece of natural language that shapes how the AI behaves — permanently, across every interaction. In Claude Code, rules live in CLAUDE.md files (project root, nested directories, or user-level). In other systems:

PlatformEquivalent ConceptWhere It Lives
Claude CodeCLAUDE.mdProject root, nested dirs, ~/.claude/CLAUDE.md
GitHub Copilot.github/copilot-instructions.mdRepository root
Cursor.cursorrulesProject root
ChatGPTCustom Instructions / MemoryUser settings (global)
GeminiGems (system instructions)Per-Gem configuration
Windsurf.windsurfrulesProject root

Why a Rule Fits Here

Conventions are persistent context. They don't change between conversations. They apply to every interaction. They're expressed naturally in prose ("use Conventional Commits") rather than code. And critically, they're advisory — the AI should follow them, but enforcement happens elsewhere. A rule that says "never commit .env files" is a strong suggestion to the AI, not a hard technical constraint. The AI can still technically commit .env files if it reasons incorrectly.

The Key Property of Rules

Rules are loaded into every conversation without being explicitly invoked. They shape behavior passively. You never type "/apply-conventions" — the AI reads the rules at session start and applies them throughout. This is why rules are the right mechanism for conventions: you want them to be omnipresent background context, not on-demand workflows.

What Rules Can't Do

Rules can't enforce anything. They're text injected into the system prompt. The model can ignore them (by mistake, not malice). Rules also can't make API calls, run code, or interact with external services. If you need a guarantee, you need a different mechanism.

3. Skills: On-Demand Prompt Workflows

Requirement #2: Create PRs properly

"When I say '/pr', check the diff against main, write a PR title under 70 characters, generate a body with Summary, Test Plan, and the Neural Blueprints footer, push the branch, and create the PR using gh CLI."

A Skill is a reusable, on-demand prompt template — a detailed instruction set that the AI follows when explicitly invoked. In Claude Code, skills are Markdown files with YAML frontmatter stored in .claude/skills/. The closest analogues in other ecosystems:

PlatformEquivalent ConceptInvocation
Claude CodeSkills (.claude/skills/*.md)/skill-name or auto-invoked by model
GitHub CopilotPrompt files (.github/prompts/*.md)Referenced in chat
ChatGPTGPTs (configured assistants)Selecting the GPT
CursorNotepads / @commandReferenced in context
GeminiGems (with specific workflows)Selecting the Gem

Why a Skill Fits Here

PR creation is a complex, multi-step workflow that you want to execute the same way every time, but only on demand. You don't want the AI creating PRs unprompted. You want a specific sequence: check diff, format title, write body, push, create. A skill encodes this sequence as a natural language prompt that guides the AI through the steps.

The critical distinction from a rule: a skill is invoked explicitly, a rule is always active. Your PR workflow isn't background context — it's a specific action triggered by a specific command. And it's more detailed than a rule should be: a rule might say "PRs should have a summary and test plan." A skill provides the exact template, the exact command sequence, and the exact formatting.

Skills Are Not Code

This is the most common misconception. A skill doesn't execute a procedure — it tells the AI how to use existing tools to accomplish a goal. The /pr skill doesn't call gh pr create directly. It instructs Claude to check the diff (using Bash), write the PR body (using reasoning), and create the PR (using Bash with gh). The AI is the executor; the skill is the playbook. This means skills degrade gracefully — if one step fails, the AI can adapt.

What Skills Can't Do

Skills are fundamentally AI-interpreted instructions. They can't guarantee execution order, can't enforce invariants, and can't run without the AI in the loop. If you need something to happen deterministically, every time, regardless of what the AI thinks, you need a hook.

4. Hooks: Deterministic Guardrails

Requirement #3: Never force-push to main

"If the AI attempts to run git push --force to main or master, block it. No exceptions. No AI judgment. Hard stop."

A Hook is a shell script that executes at a specific point in the agent's lifecycle — before a tool runs (preToolUse), after it runs (postToolUse), at session start, or at session end. Hooks are deterministic: they run real code, not prompts. They can't be persuaded, hallucinated around, or ignored.

PlatformEquivalent ConceptMechanism
Claude CodeHooks (preToolUse / postToolUse)Shell scripts in settings.json
GitHub CopilotContent Exclusion + Firewall policiesOrganization admin settings
CursorNo direct equivalent—
OpenAI CodexSandbox restrictionsContainer-level enforcement
Git (native)Git hooks (pre-push, pre-commit)Shell scripts in .git/hooks/

Why a Hook Fits Here

"Never force-push to main" is not a suggestion. It's not context. It's not a workflow. It's a hard safety constraint that must be enforced deterministically. A rule can say "don't force-push to main" — but the AI might still do it if it reasons (incorrectly) that an exception is warranted. A hook doesn't reason. It pattern-matches the command, and if it matches git push --force targeting main, it returns exit code 2 and the tool call is blocked. Period.

"Rules are suggestions to an intelligence. Hooks are laws of physics. Both are necessary, and confusing them is the most common architectural mistake in agentic systems."

The Rule-Hook Spectrum

This is the deepest insight in this entire essay. Rules and hooks exist on a spectrum of determinism:

PropertyRule (CLAUDE.md)Hook (Shell Script)
EnforcementAdvisory — model may not complyAbsolute — code-level enforcement
FlexibilityHigh — model can adapt to contextLow — pattern match only
ExpressivenessNatural language, nuancedRegex/glob, binary yes/no
When to useConventions, preferences, styleSafety constraints, compliance, audit
Failure modeModel ignores rule (soft failure)False positive blocks valid action (hard failure)

"Use Conventional Commits" → Rule. The model should follow the convention, but if a commit message is slightly non-standard, it's not catastrophic. "Never delete the production database" → Hook. You cannot afford the model getting this wrong even once. The cost of a false positive (blocking a legitimate action) is always less than the cost of a false negative (letting a dangerous action through).

5. MCPs: Structured API Access

Requirement #4: Read issues, check CI status, post comments

"I need the AI to query open GitHub issues, read pull request reviews, check the status of CI checks, and post comments — all through the GitHub API with proper authentication."

An MCP server (Model Context Protocol) is a standalone process that exposes tools to the AI agent through a standardized JSON-RPC 2.0 protocol. Unlike rules (text), skills (prompts), or hooks (shell scripts), an MCP server provides structured, typed, authenticated access to external systems.

PlatformEquivalent ConceptProtocol
Claude CodeMCP ServersJSON-RPC 2.0 over stdio/SSE
ChatGPTActions (in GPTs)OpenAPI spec → HTTP calls
GitHub CopilotExtensions / MCP support (2026)Copilot Extension API / MCP
GeminiExtensions (Google-provided)Google's proprietary extension API
OpenAI CodexMCP support (adopted 2025)MCP (same protocol as Claude)

Why an MCP Server Fits Here

GitHub's API requires authentication (tokens), returns structured data (JSON), has rate limits, has complex error handling, and has dozens of endpoints. This is fundamentally different from the other three requirements:

  • Can't be a rule — rules can't make API calls
  • Can't be a skill — skills can guide the AI to use curl or gh CLI, but they can't handle token management, pagination, error codes, or rate limiting in a structured way
  • Can't be a hook — hooks run at tool-call boundaries, not as tools themselves
  • Must be an MCP — provides typed tools with JSON Schema input/output, manages its own authentication, handles API complexity internally, and exposes clean operations like list_issues(repo, state, labels)
The Litmus Test for MCP

Ask yourself: does this require interacting with an external system's API? If yes, MCP. Rules can't make HTTP calls. Skills can guide the AI to use CLI tools, but that breaks when the CLI isn't installed, the auth token expires, or the API changes. An MCP server encapsulates all of that complexity behind a clean tool interface. The AI calls github_list_issues and gets back structured JSON — it never has to think about curl -H "Authorization: Bearer $TOKEN".

MCP vs. "Just Use the CLI"

A common objection: "Why build an MCP server when the AI can just run gh issue list?" It's a fair question. The answer is reliability and composability:

  • gh CLI requires installation and authentication setup. MCP manages its own auth.
  • CLI output is unstructured text that the AI must parse. MCP returns typed JSON.
  • CLI commands compose poorly ("list issues, then for each issue, get comments" requires the AI to write a loop in Bash). MCP tools compose through the agentic loop naturally.
  • CLI errors are strings. MCP errors are structured with error codes, enabling programmatic retry.
  • The MCP server can implement caching, rate limiting, and pagination internally — invisible to the AI.

That said, for simple cases ("just run gh pr list"), the CLI via Bash is perfectly fine. MCP shines when the integration is complex, long-lived, or shared across teams.

6. Plugins: The Ecosystem Pattern

Requirement #5: Auto-label PRs by file path

"When a PR touches files in src/auth/*, automatically add the 'security' label. When it touches src/payments/*, add the 'billing' label. This should work for any repo, not just mine."

A Plugin is the broadest concept — a third-party package that extends the AI agent's capabilities through a marketplace or registry. Plugins combine multiple mechanisms (tools, rules, resources) into a distributable unit. The concept is most mature in the ChatGPT ecosystem:

PlatformPlugin ConceptDistribution
ChatGPTGPT Store (GPTs with Actions)OpenAI marketplace
Claude CodeMCP servers (community registry)npm, pip, or standalone binaries
GitHub CopilotCopilot ExtensionsGitHub Marketplace
GeminiExtensionsGoogle-provided, limited third-party
VS Code (Copilot/Cursor)VS Code ExtensionsVS Code Marketplace

Why This One Is Different

Auto-labeling PRs by file path requires combining multiple mechanisms: an MCP tool to read the PR's changed files and apply labels (API access), a configuration file that maps paths to labels (data), and potentially a hook to trigger labeling after every PR creation (automation). No single mechanism handles this alone.

This is exactly what a plugin is: a package that bundles multiple extension mechanisms into a single installable unit. A "GitHub PR Labeler" plugin would ship an MCP server (for API calls), a config schema (for path-to-label mappings), and installation instructions to wire it into your agent's hooks or skills.

The Plugin Concept Is Converging on MCP

In practice, the "plugin" concept across AI platforms is converging on MCP as the standard protocol. OpenAI adopted MCP in 2025. GitHub Copilot added MCP support in 2026. The days of each platform having a proprietary plugin format are numbered. An MCP server is a plugin — it's just distributed as a process rather than a marketplace listing. The key remaining difference is discoverability: ChatGPT's GPT Store and GitHub's Marketplace provide discovery; MCP servers are currently discovered through word-of-mouth and GitHub repos.

8. The Decision Framework

When you're deciding how to extend your AI agent, ask these five questions in order:

QuestionIf Yes →Why
Must this be enforced deterministically, even if the AI disagrees?HookOnly hooks provide code-level enforcement. Rules can be ignored; hooks cannot.
Does this require calling an external API with authentication?MCP ServerMCPs manage auth, handle structured I/O, and encapsulate API complexity.
Is this a multi-step workflow invoked on demand?SkillSkills guide the AI through complex sequences without hard-coding the steps.
Should this apply to every conversation without being invoked?RuleRules are background context — persistent, passive, pervasive.
Does this need to bundle multiple mechanisms for distribution?PluginPlugins are composite — they combine MCP + rules + config into installable packages.

Notice the order matters. You check for hooks first because safety constraints are non-negotiable — if something must be enforced, no amount of clever prompting is a substitute for deterministic code. You check for MCP next because API access is a hard technical requirement that no other mechanism can fulfill. Skills and rules are the last resort, not because they're less important, but because they're the most flexible and should handle everything that doesn't require determinism or API access.

9. The Convergence: Why Everything Is Becoming MCP

The most important trend in AI tooling is the convergence on MCP as the universal extension protocol. OpenAI adopted MCP in 2025. GitHub Copilot added MCP support in 2026. Google's Gemini is exploring MCP compatibility. Even Microsoft's Copilot ecosystem is aligning.

This is significant because it means the "plugin" concept is standardizing. A GitHub MCP server built for Claude Code works with OpenAI Codex. A Slack MCP server built for Cursor works with Claude Code. The N x M integration problem (N AI platforms x M tools) collapses to N + M: each platform implements the MCP client once, each tool implements the MCP server once, and everything connects.

What MCP Doesn't Replace

MCP replaces Plugins and Actions — it standardizes how AI agents connect to external systems. But MCP does not replace Rules, Skills, or Hooks. These operate at different layers: Rules shape behavior (prompt-level), Skills define workflows (instruction-level), Hooks enforce constraints (code-level), and MCPs provide capabilities (protocol-level). A mature agent needs all four. MCP is the biggest single piece, but it's not the whole picture.

10. The Complete GitHub Integration

Coming full circle, here's how all five mechanisms work together for our GitHub integration:

RequirementMechanismWhy This One
Know our commit/branch/PR conventionsRule (CLAUDE.md)Persistent context, advisory, applies to all interactions
Create PRs with our exact workflowSkill (/pr)Multi-step on-demand workflow, guides AI through tools
Never force-push to mainHook (preToolUse)Hard safety constraint, must be deterministic
Read issues, check CI, post commentsMCP (GitHub server)Requires authenticated API access with structured I/O
Auto-label PRs by file pathPlugin (MCP + config)Bundles API access + configuration into a distributable package

Five requirements, five mechanisms, zero overlap. That's the point. These aren't competing solutions — they're complementary layers that handle different aspects of the same integration. The confusion arises because different platforms use different names for the same concepts, and because most tutorials show each mechanism in isolation rather than showing how they compose.

"Don't ask 'should I use a Rule or an MCP?' Ask 'which aspect of my integration requires behavioral shaping, which requires deterministic enforcement, and which requires external API access?' The mechanism follows from the answer."

The broader lesson extends beyond GitHub integration to any AI agent extension. Every time you want your agent to do something new, decompose the requirement along the three axes: determinism level, API access needs, and invocation pattern. The mechanism reveals itself. And in 2026, there's a very good chance the API access part will be an MCP server — no matter which AI platform you're building for.
